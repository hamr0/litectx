## v0.5.0 — the decision and the implementation

**Decision (user-driven, after a 3-iteration recall litmus + a competitor look):** make embeddings
**ON by default** because memory is integral to litectx and embeddings is near-essential for it.

**Implementation (surface-level, NOT a library-default flip — deliberately):**
- **CLI + MCP default to embeddings ON** (`bin/litectx.js`, `bin/litectx-mcp.js`); `--no-embeddings`
  opts out. **Library `LiteCtx` default stays `false`** → lib consumers + ALL tests + the BM25
  recall/impact/memory gates are **byte-identical** (this is why a lib-default flip was rejected: it
  breaks ~12 test files + gates, and CI runs without the optional dep).
- **Graceful fallback** (`src/index.js` `_embedSafe`): if `@xenova` can't load, disable the tier for
  that instance, warn once to stderr, continue on BM25. Wraps all 3 embed sites (index chunks,
  `_embedQuery`, `remember`). So bare installs + CI never crash.
- **`@xenova/transformers` peer→optionalDependency** (auto-installed best-effort → makes default-on
  real; install doesn't fail on native build). **Required regenerating `package-lock.json`** — the
  FIRST publish failed at `npm ci` (EUSAGE out-of-sync); fixed in `f96ac90`, tag moved, re-dispatched.
- `test/mcp.test.js` spawns with `--no-embeddings` (asserts protocol, not semantics).
- Publish = OIDC `workflow_dispatch` (`gh workflow run publish.yml --ref v0.5.0`); CANNOT `npm publish`
  locally. Run `27366887002` green; `npm view litectx version` → `0.5.0`.

## The recall litmus (committed evidence: `poc/recall-litmus*.mjs`)

Three scratch harnesses, committed as the standing evidence base:
- `recall-litmus.mjs` (litectx self-queries, naive vs LLM-expanded, BM25 vs emb sweep)
- `recall-litmus-repos.mjs` (aurora/gitdone existing labeled queries, BM25-off vs emb sweep)
- `recall-litmus-expand.mjs` (aurora/gitdone, +LLM-expansion arm)

**Findings (settled):**
- On **natural-language** code queries, embeddings give a **big, consistent lift**: aurora 0.543→0.758,
  gitdone 0.411→0.644 MRR (P@1 ~doubles, misses ~4×→1). `w=1.0` (shipped default) is best — no recalib.
- **Free LLM query-expansion recovers ~90–95%** and **erases all misses** — but it's **non-binding**
  (an agent may skip it), so embeddings is the **reliable accuracy floor**. (My iter-1 "embeddings
  marginal" read was an artifact of keyword-poor queries + POC distractors; corrected by iters 2/3.)
- **Memory (prose) embeddings is near-essential**: paraphrase recall 0.000→0.574.

## Numbers corrected this session (were inflated/mis-borrowed from aurora's torch stack)
- **Cold load:** "15–19s" was aurora's **torch** figure (mis-borrowed). litectx transformers.js/ONNX

- **Model size:** **~23 MB** (quantized ONNX), not 90 MB (fp32). Fixed in CLAUDE.md, PRD §3.3+§8,
  context.md, README, aurora-borrow-ledger (with a ⚠️ "litectx does NOT inherit" note).

## Model question — SETTLED: keep MiniLM, code-model DEPRIORITIZED
- Ours: `all-MiniLM-L6-v2` (general, 384d, ~23 MB). Theirs (DeusData/codebase-memory-mcp):
  `nomic-embed-code` (code, 768d, int8, ~7B — only viable because they **compile into a static binary**;
  NOT adoptable by a JS lib).
- **No code-specific model needed:** general MiniLM already delivered the code lift, litectx is
  **structure-first** (camelCase BM25 + import-spreading + chunk locators + impact graph + recall→impact
  disambiguation) so embeddings is a *supplementary* rerank, and a code model would risk the prose-memory
  half. `jina-embeddings-v2-base-code` is the only transformers.js-viable option but **deprioritized — no
  measured gap**; revisit only if the litmus shows one. (Recorded in the borrow-ledger.)
